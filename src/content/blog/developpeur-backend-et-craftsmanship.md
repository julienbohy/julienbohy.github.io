Je suis développeur backend PHP / Symfony.

J'ai longtemps vu mon métier comme une suite de livraisons : corriger vite, sortir les features, tenir les délais. Ça fonctionnait à court terme, mais j'avais une sensation récurrente : plus on allait vite, plus le code devenait difficile à faire évoluer.

Avec les années, cette friction a changé ma manière de travailler. Je me suis progressivement passionné pour le software craftsmanship, non pas comme un slogan, mais comme une discipline concrète.

## Ce qui a changé dans ma pratique

Au départ, je cherchais surtout des solutions techniques. Ensuite, j'ai compris que la qualité du code conditionne directement la qualité du produit et la sérénité de l'équipe.

J'ai commencé à appliquer des principes simples, de façon constante :

- clarifier l'intention du code avant d'optimiser
- réduire le couplage entre couches
- isoler les responsabilités
- automatiser les vérifications essentielles
- privilégier la lisibilité qui dure

Le résultat n'est pas seulement un code plus propre. C'est un code que l'équipe peut comprendre, discuter et faire évoluer sans dépendre de quelques personnes.

## Pourquoi le craftsmanship me parle

Le software craftsmanship m'a donné un cadre exigeant mais pragmatique.

Il ne s'agit pas de chercher la perfection. Il s'agit de livrer de la valeur avec un niveau de rigueur qui protège le futur :

- un codebase qui reste maintenable
- des décisions explicites
- une qualité partagée, pas héroïque

## Mon rôle aujourd'hui

Je reste un profil backend.

Mais je porte aussi une responsabilité sur la façon de construire :

- rendre les choix techniques plus lisibles
- élever le niveau de qualité collectif
- transformer les bonnes pratiques en habitudes d'équipe

Le code est un actif de long terme. Le craftsmanship est la manière la plus concrète que j'ai trouvée pour le respecter.
